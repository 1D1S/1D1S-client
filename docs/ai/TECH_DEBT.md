# 기술 부채 관리 프로세스

---

## 개념 정의

기술 부채는 **고이자 대출**이다. 소량씩 지속적으로 갚는 것이 폭발적으로 쌓이는 것보다 낫다.

| 구분              | 설명                                        |
| ----------------- | ------------------------------------------- |
| **아키텍처 부채** | 불변량(Invariant) 위반 — 경계가 무너진 코드 |
| **구현 부채**     | 동작은 하지만 컨벤션을 벗어난 코드          |
| **테스트 부채**   | 검증 없이 병합된 주요 로직                  |

---

## Golden Principles (스캔 기준)

AI 에이전트는 아래 항목을 기준으로 위반 여부를 판단한다.

### Critical — 즉시 수정 (PR 필수)

| 규칙                                           | 탐지 패턴                                                                |
| ---------------------------------------------- | ------------------------------------------------------------------------ |
| `'use client'`가 리프가 아닌 컴포넌트에 선언됨 | `'use client'` 선언 후 하위에 자식 컴포넌트가 다수 존재하는 Screen 레벨  |
| `useEffect` 내부에서 데이터 페칭               | `useEffect` 안에 `fetch(`, `axios.`, `apiClient.` 직접 호출             |
| `.env` 관련 값 하드코딩                        | URL, API key, secret 등 환경변수로 관리해야 할 값이 코드에 직접 기재    |
| Zod 없이 인라인 폼 검증                       | `useForm` 사용 시 `resolver: zodResolver` 없이 직접 validate 함수 사용  |

### Major — 이번 스프린트 내 수정

| 규칙                                   | 탐지 패턴                                        |
| -------------------------------------- | ------------------------------------------------ |
| TypeScript `any` 사용                  | `any` 타입 (주석 없는 경우)                      |
| `index`를 리스트 `key`로 사용          | `.map((_, i) =>` + `key={i}`                     |
| 타입 단언 (`as`) 남용                  | `as SomeType` (타입 가드 없이 단순 단언)         |
| 중첩 삼항 연산자                       | `? ... ? ... : ... : ...`                        |
| `cn()` 없이 80자 초과 `className`      | `className="..."` 줄 전체가 80자 초과 + `cn()` 미사용 |
| Query Key Factory 미사용              | 문자열 배열 직접 사용 (`['diaries', 'list']`)    |
| 캐시 무효화 누락                       | `useMutation` 성공 후 `invalidateQueries` 없음   |

### Minor — 백로그 등록 후 순차 처리

| 규칙                                | 탐지 패턴                                    |
| ----------------------------------- | -------------------------------------------- |
| 공용 컴포넌트 내부에 margin 클래스  | `app.component/` 내 `ml-`, `mr-`, `mt-`, `mb-` |
| 미사용 import                       | 실제로 참조되지 않는 import 구문             |
| 디자인 시스템 미활용                | `@1d1s/design-system`에 있는 컴포넌트를 직접 구현 |

## 스캔 방법 (AI 에이전트 수행)

```
1. 현재 브랜치의 변경 파일 또는 src/ 전체를 대상으로 위 Golden Principles 적용
2. 위반 항목을 심각도(Critical / Major / Minor)로 분류
3. 발견된 항목을 이 문서 하단에 등록
4. Critical은 즉시 수정 PR 제안
5. Major/Minor는 등록 후 우선순위 논의
```

**스캔 트리거**:

- `/ods-review` 커맨드 실행 시 — 변경 파일 대상
- 사용자가 명시적으로 "부채 스캔" 요청 시 — `src/` 전체 대상

## 정리 프로세스

```
발견 -> TECH_DEBT.md 등록 -> 심각도 분류
  ├── Critical: 즉시 수정 PR
  ├── Major: 현 스프린트 내 처리 (브랜치: refac/debt-*)
  └── Minor: 백로그 등록, 주기적 묶음 처리
```

### 브랜치 네이밍

```
refac/debt-{항목명}              # 단일 항목
refac/debt-cleanup-{날짜}       # 묶음 처리
```

### 커밋 컨벤션

```
refactor: [debt] {항목 설명}
예: refactor: [debt] useEffect 데이터 페칭 -> TanStack Query로 교체
```
